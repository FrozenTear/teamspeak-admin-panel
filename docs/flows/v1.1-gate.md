# Flow engine — v1.1 gate plan

- **Status:** draft, pending board ratification ([PURA-198](/PURA/issues/PURA-198)).
- **Companion docs:** [`architecture.md`](./architecture.md), [`http-api.md`](./http-api.md), [`ui-brief.md`](./ui-brief.md).
- **Replaces:** the V6-shaped hole in the v1.0 WS-Gate matrix (see [`docs/phase6/readiness-audit.md`](../phase6/readiness-audit.md) §1 and [`docs/deviations/v6-flow-engine-cut.md`](../deviations/v6-flow-engine-cut.md)).
- **Owner (gate harness):** QAEngineer / [QA](/PURA/agents/qa).

## 1. Goal

Add **V6 — Flow trigger** back to the Chapter 1 verification matrix for v1.1, with a fully-automated, deterministic probe that:

1. Creates a flow,
2. Enables it,
3. Manually fires it,
4. Observes the run completes successfully,

…against a fresh **rootless Podman deploy of the v1.1 release image** — the same gate posture v1.0 holds for V1–V5/V7.

A green v6-probe is the **only** condition under which `docs/deviations/v6-flow-engine-cut.md` is reclassified `resolved`.

## 2. v6-probe.sh

Path: `scripts/ws-gate/v6-probe.sh` (new directory; v1.0 did not need this folder because the existing per-verification probes lived elsewhere — v1.1 introduces the `ws-gate/` tree with v6-probe.sh as the first inhabitant so future Chapter 31 widenings have a home).

### 2.1 Inputs

```
BASE_URL    (positional 1, required)    e.g. https://manager.scuffedcrew.no
ADMIN_TOKEN (env)                       admin JWT for the manager
SERVER_CONFIG_ID  (env, default 1)
VIRTUAL_SERVER_ID (env, default 1)
OBS_WINDOW_S      (env, default 30)
```

### 2.2 Sequence (the four-step probe the issue asks for)

```bash
#!/usr/bin/env bash
# scripts/ws-gate/v6-probe.sh — V6 (flow trigger) gate probe for v1.1.
#
# 1. POST  /api/flows               create a logLine-only flow (admin)
# 2. PATCH /api/flows/{id}          set enabled=true
# 3. POST  /api/flows/{id}/fire     manual-fire it
# 4. GET   /api/flows/{id}/runs     poll within OBS_WINDOW_S until status=ok
#
# Exits 0 on success, non-zero with an error code on failure.
# All artefacts written to qa-evidence/ws-gate/v6/<timestamp>/ for matrix
# capture.

set -euo pipefail
```

Pseudocode body:

```text
1. Health pre-check
   curl -fsS "$BASE_URL/api/health"
   abort if non-2xx.

2. Create flow (admin)
   payload = {
     name: "ws-gate-v6-${random_hex8}",
     description: "WS-Gate V6 probe",
     serverConfigId: $SERVER_CONFIG_ID,
     virtualServerId: $VIRTUAL_SERVER_ID,
     enabled: false,
     definition: {
       trigger: { kind: "manualFire" },
       actions: [{ kind: "logLine",
                   message: "ws-gate v6-probe fired at ${probe_id}" }]
     }
   }
   resp = POST $BASE_URL/api/flows  (Authorization: Bearer $ADMIN_TOKEN)
   assert resp.status == 201
   flow_id = resp.body.id

3. Enable
   PATCH $BASE_URL/api/flows/$flow_id  { enabled: true }
   assert resp.status == 200 and resp.body.enabled == true

4. Manual fire
   POST $BASE_URL/api/flows/$flow_id/fire
   assert resp.status == 202
   run_id = resp.body.runId

5. Observation loop  (deadline = now + OBS_WINDOW_S, poll 500 ms)
   loop:
     GET $BASE_URL/api/flows/$flow_id/runs?limit=1
     last = body.runs[0]
     if last.id == run_id and last.status == "ok": break
     if last.id == run_id and last.status == "errored":
       fail "run errored: ${last.error}"
     if now > deadline:
       fail "run did not complete inside ${OBS_WINDOW_S}s"
     sleep 0.5

6. Cleanup
   DELETE $BASE_URL/api/flows/$flow_id?force=true
   assert resp.status == 204

7. Exit 0 with green log line.
```

All steps capture request/response bodies into `qa-evidence/ws-gate/v6/<ISO8601-UTC>/step-N.{req,resp}.json` so the matrix entry can attach them like the V1–V5/V7 evidence.

### 2.3 Exit codes

| Code | Meaning                                                                 |
| ---- | ----------------------------------------------------------------------- |
| 0    | All four steps succeeded inside the observation window.                 |
| 64   | Usage error (missing `BASE_URL` or `ADMIN_TOKEN`).                      |
| 65   | Pre-check failed (`/api/health` non-2xx).                               |
| 66   | Create failed.                                                          |
| 67   | Enable failed.                                                          |
| 68   | Fire failed.                                                            |
| 69   | Observation window expired without success.                             |
| 70   | Run errored (engine reported `errored`; body captured in evidence).     |
| 71   | Cleanup failed (test still counted green for matrix; emits a warning).  |

The non-zero discriminants let the matrix tooling distinguish "router gone again" (66) from "engine ran but errored" (70) — the latter is functionally a regression even though the wire is present.

## 5. CI / build-image integration

- The release build (`Containerfile.fullstack`) gets a `RUN scripts/check-router.sh flows_router` step that grep-fails the build if `main.rs` lacks the `flows_router` merge. Mirrors the audit's "router never merged" failure mode and prevents a regression of the v0.1.0-rc1 silent gap.
- `scripts/ws-gate/v6-probe.sh` is invoked from the existing WS-Gate runner script (whatever ran V1–V5/V7 for v1.0) as the seventh row. If the umbrella runner does not yet exist, the v1.1 gate-harness child creates one: `scripts/ws-gate/run-all.sh BASE_URL` that fan-outs to per-verification scripts and aggregates pass/fail into a single matrix dump.

## 6. Acceptance for the gate-harness implementation child

- `scripts/ws-gate/v6-probe.sh` is committed, executable, and shell-linted (`shellcheck`).
- Running the script against a locally-deployed v1.1 image exits zero and writes the seven evidence files.
- Running the script against a v1.0 image (no flows surface) fails fast on step 2 with exit 66.
